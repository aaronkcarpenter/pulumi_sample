OPA & Rego Language Reference

Comprehensive guide to Open Policy Agent (OPA) and Rego language concepts, syntax, and best practices

Table 1: OPA & Rego Language Basics

Category Feature/Concept Syntax/Example Description/Why
OPA Basics What is OPA? N/A (Conceptual) Open-source, general-purpose policy engine. Decouples policy decision-making from policy enforcement.
Core Concepts Policy, Data (Base/Virtual/Input), Query, Decision Fundamental elements of OPA's operation.
Rego Language package
#package main
#package spacelift.plan_policy
 Defines a namespace for the policy. Organizes policies and prevents naming conflicts.
import
#import data.users
#import data.acl as user_acl
#import input.user.id
 References rules/data from other packages or specific data paths. Promotes modularity and reusability.
Variables
#user := input.name
#is_valid := (status == "active")
#count = 5
 := (assignment), = (unification/comparison). Immutable once assigned in scope. Rego infers types.
Special Variables input, data, _ (wildcard) input: The document provided with the query.
data: Base and virtual documents loaded into OPA.
_: Ignores a value.
Data Types Strings "abc", 'xyz'
Numbers 10, 3.14
Booleans true, false
Null null
Arrays [1, "a"]
Objects {"key": "val"}
Sets {"a", "b"} 		Standard JSON types. Sets are unordered collections of unique primitive values.
Rego Rules Complete (Boolean)
#allow { input.method == "GET" }
 Defines a boolean outcome. Rule is true if its body (all expressions) is true. Used for simple allow/deny decisions.
Complete (Value)
#user_role := "guest" { not input.user.roles }
#max_retries = 5
 Assigns a single value to the rule. If multiple definitions can be true, the rule becomes a set of those values.
Partial (Set Gen.)
#violations[msg] { input.x < 0; msg := "x is negative" }
 Generates a set of values. Rule head rule_name[key] means rule_name will be a set where key is each item. For collecting multiple results.
Partial (Obj Gen.)
#user_details := {"name": n, "id": i} { n := ...; i := ... }
 Generates an object. Rule head rule_name := {k1:v1, ...}. For constructing structured data outputs.
default Keyword
#default allow = false
#default user_level = 0
 Provides a default value for a rule if none of its other definitions are true. Crucial for deny-by-default or ensuring a value always exists.
Rule Body { ; } Contains one or more expressions. All must be true (logical AND) for the rule to be true. Order usually doesn't matter for correctness.
else Keyword
#p { cond1 } else { cond2 } else = default_val
 Defines alternative conditions if a preceding rule body is false. Provides structured fallback logic.
Rego Ops Comparison ==, !=, <, >, <=, >= Standard comparison operators.
Assignment/Unif. := (assignment), = (unification) := strictly assigns. = asserts equality or binds variables.
Membership val in collection, key in object Checks if an element is in an array/set, or if a key exists in an object.
Arithmetic +, -, *, /, % Standard arithmetic operations.
Logical not not input.is_admin Negates an expression.
Rego Control Iteration
#item := array[_]
#val := object[key]
#elem := set_elements[_]
 Loops over arrays (values via _), objects (keys via key var, then access object[key]), or sets (values via _).
Functions (User)
#is_valid_user(user) { user.active }
#allow { is_valid_user(input.user) }
 Reusable logic blocks that take arguments. Improves policy organization and reduces redundancy.
Built-in Functions count(array), sprintf(format, args), object.get(obj, key, def) Rich library for common operations (strings, arrays, objects, types, encoding, time, crypto, HTTP, regex). Refer to OPA docs for full list.
Comprehensions Array: [term | body]
Set: {term | body}
Object: {key:val | body} 		Concise way to build arrays, sets, or objects based on iteration and conditions.
with Keyword
#allow with input.user.role as "admin"
#data.servers[_] with data.ports as mock_ports
 Temporarily replaces parts of data or input for a specific evaluation. Essential for testing and "what-if" scenarios.
OPA is an open-source, general-purpose policy engine that enables unified, context-aware policy enforcement across the stack

Table 2: Writing & Testing OPA Policies

Category Feature/Concept Syntax/Example Description/Why
Policy Writing Allow/Deny Structure
#default allow = false
#allow { condition1 }
#allow { condition2 }
 Common pattern: deny by default, explicitly define conditions for allow. Multiple allow rules act as OR.
Violation Messages
#deny[msg] { condition; msg := "Error: ..." }
#Query: data.package.deny
 Instead of boolean, return a set of messages detailing reasons for denial. More informative for users/systems.
Working with input
#input.request.user.id
#input["resource-type"]
#object.get(input, "optional_field", default_val)
 input is the query data. Access nested fields. Use object.get for safe access to potentially missing fields.
Policy Modularity
#File 1: package common.utils
#File 2: package app.policy; import data.common.utils
 Break complex policies into smaller files/packages. Use import to combine logic for better organization and reusability.
Testing Policies opa test Command
$opa test . -v
$opa test ./policies/ --coverage
 Command-line tool to run tests. -v for verbose output. --coverage for test coverage report.
Test Files (_test.rego)
#package mypolicy # Same package or import
#test_my_rule_passes { mypolicy.my_rule with input as ... }
 Test rules start with test_. Typically in files ending with _test.rego. Use with to mock input and data.
Assertions
#test_is_true { rule_being_tested }
#test_is_false { not rule_being_tested }
#test_value { res := rule; res == "expected" }
 Test passes if the test rule evaluates to true. For specific value checks, assign rule output and compare.
trace() for Debugging
#trace(sprintf("Value: %v", [my_var]))
 Prints messages to stderr during evaluation. Useful for debugging complex rules within tests or during development.
Tip: Use the opa test command with mock input JSON files that replicate Spacelift's data for robust testing

Example Policies Plan Policy: S3 Versioning Denies S3 buckets without versioning enabled when created or updated.
#package spacelift
#deny[reason] {
# change := input.spacelift.terraform.resource_changes[_];
# change.type == "aws_s3_bucket";
# not change.change.actions[_] == "delete";
# not object.get(change.change.after, "versioning", {}).enabled == true;
# reason := sprintf("S3 bucket '%s' must have versioning enabled.", [change.address])
#}
Approval Policy: Prod Access Requires manual approval for changes to production stacks by non-admins.
#package spacelift
#approve_manually[reason] {
# input.spacelift.stack.labels[_] == "env:prod";
# not input.spacelift.run.user.role == "admin";
# reason := "Changes to production by non-admins require manual approval."
#}

Testing Use opa test with mock input JSON files that replicate Spacelift's data. -
Modularity Leverage multiple Rego files and packages for complex policies; Spacelift supports this. -
Spacelift policies enable granular control over infrastructure changes, security, and compliance

Table 4: Advanced Rego & Best Practices

Category Feature/Concept Description/Why
Advanced Rego Recursive Rules
#is_ancestor(a, b) { parent_of(a, b) }
#is_ancestor(a, c) { parent_of(a, b); is_ancestor(b, c) }
Why? Solves problems like transitive closure.
every Keyword (or equivalent)
#all_active { every u in data.users { u.status == "active" } }
Tip: Often implemented with count(violations) == 0 for current OPA versions.
Partial Evaluation OPA pre-computes parts of a decision when some data is known at policy load time.
Tip: Write clear, modular Rego to allow for optimal engine performance.
Best Practices Start Simple Begin with clear, straightforward rules and build complexity incrementally.
Tip: Avoid over-engineering initially; focus on core requirements.
Deny by Default For authorization policies, it's safer to implicitly deny and explicitly allow.
Tip: default allow = false is a common and secure pattern.
Descriptive Naming Use clear, concise names for packages, rules, and variables.
Tip: is_s3_encrypted is better than check_s3.
Small, Focused Policies Break down large policies into smaller, reusable modules.
Tip: Use separate files/packages for different policy concerns (e.g., s3_policy, ec2_policy).
Thorough Testing Write comprehensive tests for all policy logic, including edge cases.
Tip: Use opa test with mocked input and data for robust validation.
Version Control Store your policies in a Git repository.
Tip: Treat policies as code; use branches, pull requests, and commit messages.
Iterate and Refine Policy development is often an iterative process of testing and refinement.
Tip: Don't aim for perfection on the first try; continuously improve policies based on feedback and new requirements.
Use Rego Playground Excellent tool for experimenting, debugging, and sharing Rego snippets.
Tip: play.openpolicyagent.org for quick testing and validation.
Understand input Schema Deeply familiarize yourself with the structure of your input data.
Tip: The exact input schema provided by Spacelift (or your system) is the foundation of effective policies.
Avoid Overly Complex Rules If a rule becomes hard to understand or test, refactor it into smaller functions or rules.
Tip: Prioritize readability and maintainability over single-line "clever" solutions.
Resources Official OPA Documentation The primary and most authoritative source for OPA and Rego.
Spacelift OPA Policy Docs Specific documentation on Spacelift's input schemas and policy evaluation behavior.
Rego Playground Online environment for writing, testing, and sharing Rego code.
Community/Forums Engage with the OPA community for questions and best practices.
